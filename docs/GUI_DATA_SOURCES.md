# GUI Data Sources

The Oracle GUI labels major sections by source so operators can distinguish live backend state from saved validation evidence and static placeholders.

## Source Labels

- `LIVE`: Current backend endpoint or service health.
- `REPORT`: Saved validation or capability report.
- `LIVE/REPORT`: Live endpoint backed by a current or latest generated report.
- `LIVE/CONFIG`: Runtime configuration or scheduler state.
- `LIVE SAFETY POLICY`: Runtime safety policy enforced by disabled controls.
- `LOCKED`: Production or destructive control blocked in the GUI with a visible reason.
- `DEMO`: Static placeholder until live data stream is connected.

## Dashboard

- Backend status: `LIVE`
- Service health: `LIVE`
- Latest latency/status: `REPORT`
- Evolution metrics: `REPORT`
- Warnings: `LIVE/REPORT`
- Attack timeline: `DEMO`
- Monitoring mode: `LIVE/REPORT`; live network capture is `ACTIVE` only when sensor readiness passes and capture is running.

## MutantShield

- Model load state: `REPORT` unless a dedicated live model endpoint is added.
- CSE/DoHBrw metrics: `REPORT`
- Evolution dry-run action: `LIVE`
- Production retraining/promotion: `LOCKED`
- Monitoring mode: `LIVE/REPORT`

## QAuthCore

- Health/token status: `LIVE` if service health endpoint is reachable.
- Capability metrics: `REPORT`
- User management: `LOCKED`
- Placeholder session counts: `DEMO`

## ChronoLedger

- Health/chain status: `LIVE`
- Append/validation metrics: `REPORT`
- Append/export actions from GUI: `LOCKED`
- Blockchain visualization rows: `DEMO`

## GhostTunnel

- Health/jobs: `LIVE/REPORT`
- Capability metrics: `REPORT`
- Create persistent tunnel: `LOCKED`
- Demo transmit test: `LIVE`
- Topology/tunnel rows: `DEMO`

## Evolution Engine

- Scheduler state: `LIVE/CONFIG`
- Candidates/metrics: `REPORT`
- Promotion controls: `LIVE SAFETY POLICY`
- Promotion controls are also labelled `LOCKED` in operator language because production models remain unchanged.

## Latest Events

Latest events are read from live proof reports and ChronoLedger/evolution evidence summaries. Realtime replay proof entries are labelled `LIVE_REPLAY`; live sensor fallback entries are labelled through the replay proof report.

## Monitoring Truth

The GUI distinguishes:

- `LIVE NETWORK CAPTURE`: only active when Scapy/Npcap/permissions and sensor readiness pass.
- `LIVE REALTIME REPLAY`: validated safe proof generated by `python scripts/oracle_realtime_replay_proof.py --events 25`.
- `REPORT SNAPSHOT`: saved validation/capability evidence.

If live packet capture is unavailable, the correct operator message is: `Live network capture is not active. Realtime replay proof is available and validated.`
